## DAY 3 — WHAT TO BUILD

### Phase 1 — LLM Question Generation Per Candidate

Create `backend/app/services/question_generator.py`.

**Function:** `async def generate_questions(job: Job, candidate: Candidate) -> list[dict]`

- Calls Gemini 2.5 Flash (same client pattern as `jd_analyzer.py` — use `google-genai` SDK, not `google-generativeai`)
- Prompt must instruct Gemini to generate exactly **5 interview questions** tailored to both the job's `required_skills`, `role_level`, `implicit_signals` AND the candidate's `current_title`, `listed_skills`, `experience_summary`
- Questions must vary in type: at least 1 behavioural, at least 1 technical, at least 1 situational
- Response must be a JSON array of exactly this shape:
```json
[
  {
    "id": 1,
    "type": "technical|behavioural|situational",
    "question": "string",
    "time_limit_seconds": 120
  }
]
```
- Same retry logic as `jd_analyzer.py`: attempt 1 with primary prompt → on JSON failure attempt 2 with strict prompt → on second failure raise `HTTPException(500)`
- `time_limit_seconds` must be set by Gemini based on question complexity — not hardcoded

**Add new endpoint to `backend/app/routers/interview.py`:**

```
POST /interview/{token}/questions
```

Behaviour:
- Look up candidate by `interview_token` → 404 if not found
- Verify `interview_status == "in_progress"` → 400 if not (message: `"Interview must be started before fetching questions"`)
- Call `generate_questions(job, candidate)`
- Store the result as `questions` JSONB on a new `InterviewSession` model (see Phase 2)
- Return the list of questions

### Phase 2 — InterviewSession Model

Create `backend/app/models/interview_session.py`.

**Columns:**
```
id                UUID, primary key
candidate_id      UUID, FK → candidates.id CASCADE
job_id            UUID, FK → jobs.id CASCADE
questions         JSONB — list of question dicts generated by Gemini
answers           JSONB — list of {question_id, transcript, video_url}, default []
started_at        DateTime(timezone=True), nullable
completed_at      DateTime(timezone=True), nullable, default None
```

- One session per candidate per job (enforce with a unique constraint on `candidate_id + job_id`)
- Add relationship to Candidate: `interview_session = relationship("InterviewSession", back_populates="candidate", uselist=False)`
- Import it in `backend/app/models/__init__.py` so Alembic detects it
- Generate and apply Alembic migration: `alembic revision --autogenerate -m "add_interview_session"` then `alembic upgrade head`

**Update `POST /interview/{token}/questions`:**
- Check if a session already exists for this candidate+job — if yes, return the existing questions without calling Gemini again
- If no session exists, generate questions, create the `InterviewSession` record with `started_at = datetime.now(timezone.utc)`, persist, return questions

### Phase 3 — Chunked Video Upload with Resume-on-Failure

**Backend — new endpoint in `backend/app/routers/interview.py`:**

```
POST /interview/{token}/upload-chunk
```

Request: `multipart/form-data` with fields:
- `question_id: int`
- `chunk_index: int`
- `total_chunks: int`
- `video_chunk: UploadFile`

Behaviour:
- Look up candidate by token → 404 if not found
- Upload the chunk to Supabase Storage at path: `interviews/{candidate_id}/{question_id}/chunk_{chunk_index}.webm`
- Use `SUPABASE_URL` and `SUPABASE_KEY` from environment (already in `.env.example`)
- Use `httpx` to call the Supabase Storage REST API directly — do not add a new Supabase SDK dependency
- Return: `{ "chunk_index": int, "received": true }`

```
POST /interview/{token}/finalize-answer
```

Request body:
```json
{ "question_id": 1, "total_chunks": 5 }
```

Behaviour:
- Look up candidate by token → 404 if not found
- Fetch the `InterviewSession` for this candidate
- Construct the final video URL pattern: `{SUPABASE_URL}/storage/v1/object/public/interviews/{candidate_id}/{question_id}/chunk_0.webm` (Day 4 will handle merging — for now store chunk_0 URL as a placeholder)
- Append `{ "question_id": question_id, "transcript": null, "video_url": <constructed_url> }` to `session.answers`
- Persist the updated session
- Return: `{ "question_id": int, "video_url": str, "answers_submitted": int }`

### Phase 4 — Mobile Recording Interface

Replace the stub at `mobile/src/app/interview/[token]/questions.tsx` with the full recording screen.

**Behaviour:**
- On mount: call `POST /interview/{token}/questions` to fetch the 5 questions
- Show one question at a time with a question counter ("Question 1 of 5")
- Show the question `type` as a badge (Technical / Behavioural / Situational)
- Show a countdown timer for `time_limit_seconds` — auto-advance to submission when timer hits 0
- **Recording:**
  - Use `expo-av` for video recording (`Camera` + `Audio` permissions)
  - Start recording when user taps "Start Recording"
  - Stop recording when user taps "Stop Recording" or timer expires
  - On stop: chunk the recorded video into pieces of max 2MB each and upload each chunk via `POST /interview/{token}/upload-chunk`
  - Show upload progress (e.g. "Uploading chunk 2 of 4...")
  - On all chunks uploaded: call `POST /interview/{token}/finalize-answer`
- After finalizing answer for question N, show "Next Question" button (or "Submit Interview" on question 5)
- On final submission: navigate to `mobile/src/app/interview/[token]/complete.tsx`

Create `mobile/src/app/interview/[token]/complete.tsx`:
- Simple screen: "Interview Complete. Thank you, {candidate_name}. Your responses have been submitted."
- No further navigation

**Permissions:** request Camera and Microphone permissions at the start of the questions screen before attempting to record. If denied, show an error message explaining permissions are required.

**Install required Expo packages:**
```
npx expo install expo-av expo-camera
```
